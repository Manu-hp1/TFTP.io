PROTOCOL OVERVIEW

How TFTP Works

TFTP runs over UDP on port 6969. Every file transfer is a handshake sequence — a request, data blocks of 512 bytes each, and an acknowledgement for every single block.

STEP 01
CONNECT to Server
Creates UDP socket, validates server IP using inet_pton() and checks port range 1024–65535. Stores server address in client struct for all PUT and GET operations.
socket(AF_INET, SOCK_DGRAM, 0)
STEP 02
Send & Receive Packets
PUT sends WRQ then DATA blocks. GET sends RRQ and receives DATA. A do-while on recvfrom() verifies sender IP and port — prevents packet mixing from multiple clients.
sendto() · recvfrom() · htons()
STEP 03
Queue-Based Server
Server binds to INADDR_ANY on port 6969. No fork() — one client served at a time using the UDP kernel receive buffer as a natural FIFO queue.
bind() · INADDR_ANY · waitpid()
FILE TRANSFER OPERATIONS

PUT & GET Handshake

PUT uploads a file client→server. GET downloads server→client. Both follow a strict block-by-block handshake until an empty DATA packet signals end of file.

PUT — WRQ (Write Request)
1
Client sends WRQ with filename and mode
pkt.opcode = htons(WRQ)
2
Server replies ACK 0 — ready to receive
recvfrom() → ACK block 0
3
Client sends DATA block 1 (512 bytes)
pkt.opcode = htons(DATA)
4
Server ACKs every block received
sendto() → ACK block N
5
Client sends empty DATA (size=0) → EOF
data_size = 0 → EOF signal
UDP · PORT 6969
GET — RRQ (Read Request)
1
Client sends RRQ with filename and mode
pkt.opcode = htons(RRQ)
2
Server sends DATA block 1 directly — no prior ACK
sendto() → DATA block 1
3
Client ACKs each block and writes to file
write(fd, data, size)
4
Server sends next block on each ACK
recvfrom() → ACK N → send N+1
5
Server sends empty DATA → EOF signal
if (data_size == 0) → EOF

KEY FEATURES

What MANUSHARE Does

🖥️
Interactive CLI — MANUSHARE
Custom colored terminal menu with CONNECT, PUT, GET, MODE CHANGE and EXIT options. Real-time block-by-block transfer logs showing opcode, block number and transfer size on every packet sent or received.
CONNECT → PUT / GET → MODE → EXIT
🔗
CONNECT Operation
Creates UDP socket, validates server IP with inet_pton(), checks port range 1024–65535. Must connect before any PUT or GET.
🛡️
Packet Collision Guard
do-while on recvfrom() verifies sender IP and port match current client. Silently discards packets from other clients.
📤
PUT — File Upload
WRQ → ACK 0 → DATA blocks → EOF. Sends empty DATA packet as end-of-file signal after last block.
📥
GET — File Download
RRQ → DATA blocks → ACK each. Receives directly without waiting for prior ACK. Stops on empty DATA (EOF).
TECHNOLOGY STACK

Built With

LANGUAGE01
C Programming
Pure C with no external libraries. Every socket, packet, file operation and mode conversion built from scratch using standard POSIX calls.
PROTOCOL02
UDP / TFTP RFC 1350
SOCK_DGRAM over UDP port 6969. Implements RRQ, WRQ, DATA, ACK and ERROR packets with correct opcode encoding using htons().
NETWORK03
Socket Programming
socket(), bind(), sendto(), recvfrom(), inet_pton(), htons(), ntohs() — full POSIX networking stack for UDP client-server communication.
SYSTEM04
Linux System Calls
open(), read(), write(), close(), lseek() for file I/O. INADDR_ANY for server binding. Kernel UDP buffer for queue-based client handling.
CONVERSION05
NETASCII Mode
netascii_linux() strips \r for Linux targets. netascii_windows() converts \n to \r\n. Separate write index j prevents the buffer overwrite bug.
PLATFORM06
Linux / GCC
gcc -o server tftp_server.c tftp.c and gcc -o client tftp_client.c. Compiled with -Wall -Wextra for zero warnings on Linux WSL.
